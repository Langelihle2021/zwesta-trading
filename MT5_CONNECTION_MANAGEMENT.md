# MT5 Connection Management Guide

## Overview
This document explains how MT5 connections are managed in the Zwesta Multi-Broker Backend, including connection lifecycle, caching mechanisms, and proper cleanup procedures.

## Connection Management Architecture

### Global Components

#### 1. MT5 Connection Lock (`mt5_connection_lock`)
- **Purpose**: Prevents simultaneous MT5 connections which cause IPC conflicts
- **Type**: `threading.Lock()`
- **Usage**: Acquired before any MT5 connection attempt
- **Timeout**: Configurable per operation (typically 30-60 seconds for full connections, 2 seconds for balance checks)

```python
# Example usage in MT5Connection.connect()
lock_acquired = mt5_connection_lock.acquire(timeout=float(lock_timeout))
if lock_acquired:
    try:
        # Connection logic here
    finally:
        mt5_connection_lock.release()
```

#### 2. Balance Cache (`balance_cache`)
- **Purpose**: Caches account balances to return when MT5 connection unavailable
- **Type**: Dictionary with format `{"{broker}:{account}": {'balance': X, 'equity': Y, ...}}`
- **Thread-Safety**: Protected by `balance_cache_lock`
- **Keys Generated By**: `get_balance_cache_key(broker_name, account_number)`
- **Populated On**:
  - Backend startup (hardcoded demo accounts)
  - Successful MT5 login
  - Balance API endpoint calls

**Cache Key Auto-Detection** (prevents mismatches):
- Exness accounts (298997455, 295619855) mapped explicitly
- Falls back to normalized broker name + account number
- Ensures consistent keys across code

#### 3. Broker Manager (`broker_manager`)
- **Purpose**: Maintains persistent connections for API operations
- **Type**: `BrokerManager` class instance (global)
- **Storage**: `broker_manager.connections` dictionary
- **Key Format**: Account ID (string, e.g., "Exness MT5")

### Connection Cache for Bots (`broker_connection_cache`)
- **Purpose**: Caches MT5 connections temporarily per bot to avoid repeated connection overhead
- **Type**: Dictionary with format `{"{user_id}|{broker}|{account}": connection_object}`
- **Thread-Safety**: Protected by `broker_connection_cache_lock`
- **Lifetime**: Cleared when bot stops or backend restarts
- **Savings**: 3-5 seconds per trade when using cached connection

## Connection Lifecycle

### 1. Login/Test Connection (`/api/broker/test-connection`)

```
User provides credentials
    ↓
Create temporary MT5Connection object
    ↓
Acquire mt5_connection_lock
    ↓
Initialize MetaTrader5 module (if needed)
    ↓
Connect with provided credentials
    ↓
Get account info & balance
    ↓
Update balance_cache with real balance
    ↓
Disconnect (release MT5 terminal)
    ↓
Save credentials to database
    ↓
Return balance to frontend
```

**Important**: Temporary connection is NOT stored in `broker_manager.connections`

### 2. Logout (`/api/broker/exness/logout`)

```
User initiates logout
    ↓
Identify all Exness connections in broker_manager.connections
    ↓
For each connection:
    - Call connection.disconnect() (calls mt5.shutdown())
    - Remove from broker_manager.connections
    - Clear associated balance_cache entries
    ↓
Shutdown global MetaTrader5 module (safety)
    ↓
Return success
```

**New Functionality**:
- Properly removes connections from `broker_manager.connections`
- Clears balance cache entries
- Logs cleanup details
- Returns number of accounts cleaned

### 3. Bot Trading

```
Bot creation → Load credentials from database
    ↓
check broker_connection_cache for existing connection
    ↓
If found & active: Use cached connection (saves 3-5s)
    ↓
If not: Create new MT5Connection
    ↓
Acquire mt5_connection_lock
    ↓
Connect to MT5
    ↓
Update balance_cache with real balance
    ↓
Execute trading loop (keeps connection active)
    ↓
On bot stop: Clear broker_connection_cache
```


This prevents cache misses when:
- Database stores "MetaTrader 5" but code uses "Exness"
- Different parts of code refer to same account by different names

## Common Issues & Solutions

### Issue: "MT5 already initialized" error
**Cause**: Previous connection not properly cleaned up
**Solution**: Call `mt5.shutdown()` before attempting new connection

### Issue: Balance shows $10,000 (default) instead of real balance
**Cause**: 
1. MT5 connection lock timeout (another operation holding lock)
2. Balance cache key mismatch (different broker name format)
3. Account not in cache and MT5 unavailable
**Solution**: Check `balance_cache` keys and ensure login updates cache

### Issue: Users can't reconnect after logout
**Cause**: Connection still in `broker_manager.connections` but MT5 module shut down
**Solution**: Use new `remove_connection()` method to properly clean up

### Issue: Memory leak with multiple connections
**Cause**: Connections created but never removed from `broker_manager.connections`
**Solution**: Ensure all logout paths use `broker_manager.remove_connection()`

## Thread Safety Considerations

### Locks in Order of Acquisition (prevent deadlock)
1. `balance_cache_lock` - Always acquire first (caches)
2. `mt5_connection_lock` - Then acquire for MT5 operations
3. `broker_connection_cache_lock` - Then acquire for broker cache (bot-specific)

### Common Operations & Locks
```python
# Login test (gets real balance)
with mt5_connection_lock:
    with balance_cache_lock:
        # Update balance cache after login

# Bot trading
with broker_connection_cache_lock:
    # Check/use cached connection
# Then with mt5_connection_lock:
    # Connect if needed

# Logout
# Already has user context, no global lock needed initially
# Then with balance_cache_lock:
    # Clear cache entries
```

## Migration from Old Code

If updating existing deployments:

1. **Verify balance_cache keys**: Check for mismatches like "MetaTrader5:298997455" vs "Exness:298997455"
2. **Run connection cleanup**: Reset or clean out `broker_manager.connections` if stale
3. **Test logout**: Ensure new `remove_connection()` method works
4. **Monitor logs**: Watch for "Removed connection" messages during testing

## Performance Implications

### Connection Caching Benefits
- **Test Connection**: ~0.5-2 seconds per test (full connect)
- **Bot Trading with Cache**: ~0.1 seconds per trade (using cached connection)
- **Savings**: 3-5 seconds per trade when using cached connection

### Balance Cache Benefits
- **No Cache Hit**: MT5 lock timeout → $10,000 default
- **Cache Hit**: Return in <0.1 seconds
- **Network Latency**: ~0.5-2 seconds for fresh MT5 query
